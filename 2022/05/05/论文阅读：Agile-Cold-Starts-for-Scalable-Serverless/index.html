论文阅读：Agile Cold Starts for Scalable Serverless

Agile Cold Starts for Scalable Serverless

引用格式：Mohan A, Sane H, Doshi K, et al. Agile cold starts for scalable serverless[C]//11th USENIX Workshop on Hot Topics in Cloud Computing (HotCloud 19). 2019.

出处：2019-workshop

一、Abstract

提出问题：

​ 在Serverless或者Faas架构中会存在两个问题：

  1. 缓存或者重复使用以前使用过的容器会占用内存并且有信息泄漏的风险。
  2. 冷启动延时。

解决问题：

​ 预先制作这些资源，然后动态地将它们与基线容器重新关联。

实验效果：

冷启动方面有一个数量级的提高，内存消耗可以忽略不计，在并发量增加的情况下启动时间持平。

二、Introduction

问题具体描述：

冷启动问题：函数执行必须依赖容器的启动，在Serverless中，通常是按需启动容器。于是存在冷启动的问题。并且冷启动的时间随着 并发性的增加而急剧上升。

造成冷启动时间增加的原因：网络创建和初始化。

解决方案：

  1. 以OpenWhisk（Serverless框架）和Pause容器为基础。提出Pause Container Pool Manager（PCPM）将冷启动延迟及其并发影响降低了近一个数量级。在一百个并发的容器中，启动时间减少了近80%，而消耗的系统内存却可以忽略不计。
  2. 预先创建并缓存网络端点。
  3. 在创建时将其与功能容器绑定。
  4. 在功能容器拆除时将其分离出来以便重用。

Pause Container作用：

在K8S里面，一个Pod里包含多个容器，Puase容器用于解决如何让一个 Pod 里的多个容器之间最高效的共享某些资源和数据。

相当于一个代理模式。

三、Related Work

pre-warmed containers方法：

预热容器，预先创建带有所需代码及其依赖关系的容器并在调用时启动它来节省启动一个新容器的成本。

warm container：

通过缓存一个具有所需环境的容器的运行副本，来节省启动时间。

周期性预热：

每隔一段时间提交假的请求，以保持容器的温度。

不足：

  1. 过度配置这些资源来浪费内存、交换空间
  2. 轻微的供应不足也会引起冷启动突发，并产生大量的启动时间
  3. 不能有效地缓解非确定性和并发性的调用

五、网络对冷启动时间的影响

一个容器经历了四个主要阶段

  1. 服务调用，其中一个功能触发器到达Docker守护进程
  2. 启动，包括创建一个容器，设置其网络，并将其连接到网络
  3. 运行时间，其中功能被执行
  4. 清理，包括停止容器，断开其网络连接，并销毁它。

服务调用、启动和运行时间直接落在函数执行的关键路径上。清理步骤可能间接地落在其他功能的关键路径上，因为它需要Docker守护程序的周期。

从图可以看到，启动时间的增长是限制性能扩展的主要因素，第二因素是容器的清理时间。进一步来说网络创建和网络连接步骤占了启动时间的90%。具体实验的结果如下图所示。

六、Pause Container Pool Manager (PCPM)

模型介绍：包括预先创建网络并将其连接到功能容器上，原本花在网络创建和网络连接步骤上的时间就可以从启动的关键步骤中移除。

6.1Network Pre-creation

提出了pause container（PC）池化概念。

  1. PC是提前创建的，其初始化在网络创建步骤之后暂停，在该步骤中为其分配一个IP地址。
  2. 一个正常的Docker（或任何）容器随后在需要时被连接到PC上，从而有效地共享其预先建立的网络命名空间。
  3. 由于PC是网络的一部分，任何附着在PC上的容器也是同一网络的一部分，并且可以与属于该网络的其他容器进行通信。
  4. PCPM创建PC并将它们放在一个池中。当需要时，它们就会从池中取出，并与功能容器绑定；而当这些应用容器准备终止时，PC就会被分离并放回池中。
  5. PC与环境无关，可以被任何应用容器使用

6.2PC Pool Management

PCPM管理着一个PC池；它在三个关键阶段进行管理。

  1. 如图（a）所示，在构建阶段，一些暂停的容器被启动。这时，PCPM用对应于PC的标识符初始化一个自由池。调用者通过PCPM掌握了关于PC的知识。
  2. 图（b）显示了第二个阶段，即执行阶段。在这一阶段，调用者查询池管理器并获得可用PC的标识符（例如，图（b）中的PC 1）。使用这个标识符，调用者将新启动的容器附加到相应的PC（即PC 1）上，并将其从空闲池中移除。
  3. 图（c）描述了第三个阶段，即完成阶段，当一个函数完成后，其执行容器被终止或以其他方式回收。

七、Result

基于PCPM的OpenWhisk的并发执行情况，使用未修改的OpenWhisk作为基线。在这一节中，我们与预热和温热容器的缓解措施进行了比较，结果显示执行时间节省了80%（相对于冷启动），相对于预热/温热容器，内存节省了几个数量级。

八、Summary and Discussion

本文提出的解决方案预先创建网络（而不是预先创建容器），并将预先创建的网络无缝地附加到功能容器上。与冷容器相比，执行时间减少了80%，与预热容器相比，内存占用减少了几个数量级，性能下降不大。

需改进的地方：

  1. 使用IP地址，会带来安全问题。我们希望在网络命名空间上增加多种安全和隔离策略，以加强PCPM的安全方面。
  2. 预先创建或预先获取资源的想法可以扩展到网络实体之外，比如预先创建功能环境的组件和预先获取代码依赖，以使FaaS以最小的内存占用率敏捷地进行。
